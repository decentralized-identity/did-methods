# DID method standardization proposal: did:ethr

This is a proposal to include `did:ethr` in the set of DID methods that will be standardized by the DID Methods WG.

## Description

`did:ethr` is a DID method based on an Ethereum Virtual Machine (EVM) blockchain, that utilizes the ERC-1056 lightweight
identity registry smart contract. It allows any Ethereum address, smart contract, or secp256k1 public key to become a
valid decentralized identifier with no on-chain registration required.

### Benefits

- **Zero-cost creation**: No transaction fee or on-chain registration needed to create an identifier. Any Ethereum
  public key, address, or smart contract is implicitly a valid DID.
- **Private creation**: Identifier creation happens off-chain and is completely private.
- **Multi-network support**: Deployed on Ethereum mainnet, Gnosis, Polygon, Energy Web Chain, Sepolia, and other
  EVM-compatible chains. Supports any EVM-compatible blockchain with a deployed registry.
- **Key rotation**: Full support for changing the controller of a DID via the ERC-1056 `changeOwner` function.
- **Multiple Verification Methods**: Supports Secp256k1 public keys by default in the genesis DID document. Any other
  Verification Method can be added through an update, even if not specifically mentioned in the spec.
- **Service endpoints**: Supports adding arbitrary service endpoints.
- **Meta-transactions**: Supports externally signed transactions, allowing third-party gas funding without giving away
  custody or control of a DID and without risk of censorship.
- **Multi-sig / proxy support**: Controller can be set to a smart contract, enabling multi-signature or proxy control,
  per DID, through any logic that can be encoded as smart contract code.
- **Account abstraction**: Leverages Ethereum's built-in account abstraction.
- **Verifiable versioning**: Supports resolving historical versions of DID documents via `versionId` query parameter.
- **Deactivation**: Supports DID deactivation by setting the owner to the null address.

### Drawbacks

- **Transaction fees for updates**: While DID creation is free, any update (controller rotation, verification method or
  service endpoint update) requires an on-chain transaction and associated gas fees.
- **Secp256k1-native for on-chain operations**: DID document updates are normally signed with secp256k1 keys. Support
  for other key types as controllers is possible but requires the use of additional smart contracts, which may be less
  efficient.
- **Network dependency**: Resolution requires access to an Ethereum RPC endpoint for the target network.
- **Gas price volatility**: Transaction costs are exposed to the gas price of the target network where a DID is
  anchored.
- **Not self-certifying**: The DID identifier does not embed a commitment to the full DID document; it is derived from
  the Ethereum address or public key. The spec defines a self-certifying Genesis DID document. Updates follow a
  cryptographically verifiable history.

## Is this DID method already involved in a standardization process? If so, where?

The `did:ethr` method is registered in the
[W3C DID Specification Registries](https://www.w3.org/TR/did-spec-registries/#did-methods).
The underlying smart contract interface was proposed as
[ERC-1056](https://github.com/ethereum/EIPs/issues/1056) in the Ethereum
community. The reference resolver is maintained under the
[Decentralized Identity Foundation](https://github.com/decentralized-identity/ethr-did-resolver)
GitHub organization. No formal standardization working group has previously
been dedicated to `did:ethr` specifically.

## Supporting use cases

All [DID Use Cases](https://www.w3.org/TR/did-use-cases/) are supported by this DID Method. This DID Method is
specifically beneficial for the following classes of use cases:

- **Natural persons**: Any individual holding an Ethereum private key has an implicit `did:ethr` identifier with no
  registration step. This enables seamless integration with VC wallets and decentralized identity applications built
  on the Ethereum ecosystem, while preserving the option to keep the DID entirely off-chain.
- **Legal entities and organizations**: Organizations can use smart contract-controlled DIDs with multi-signature
  governance policies enforced on-chain, enabling team-managed organizational identities with configurable signing
  thresholds and programmable access control without relying on any central authority.
- **Verifiable credential issuers**: Institutions issuing long-lived credentials (diplomas, licenses, professional
  certifications) benefit from DIDs that are permanent, support key rotation, and retain a cryptographically
  verifiable history of all key changes — ensuring issued credentials remain verifiable even after key updates.
- **Machine identity and IoT**: Devices and automated agents with secp256k1 keys can be assigned permanent,
  manageable identities without centralized provisioning infrastructure. Delegate keys with configurable expiry
  enable temporary operational access without modifying the root DID controller.
- **Decentralized applications (dApps)**: Every Ethereum address is already a `did:ethr` identifier, making it the
  natural identity layer for dApps. No additional identity infrastructure is required to issue or verify credentials
  tied to on-chain participants.
- **Energy sector and sustainability**: The Energy Web Chain deployment enables green energy certificate issuance,
  grid device identity management, and participation in the EW-DOS ecosystem using `did:ethr` as the identity layer.
- **Gasless / sponsored identity management**: The built-in meta-transaction support in the ERC-1056 registry, and
  compatibility with broader Ethereum account abstraction standards (ERC-4337, EIP-7702), enables third parties to
  cover on-chain costs on behalf of DID controllers without ever gaining custody or control of the DID.
